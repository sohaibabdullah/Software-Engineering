1. FIRST, TDD, and Red–Green–Refactor

1.1 From Validation to Verification

Module 2 distinguished two aspects of software assurance:

Validation Verification
"Did you build the right thing?" "Did you build the thing right?"

This module focuses on verification — building the thing right — via software testing as part of the Agile lifecycle. Although testing is only one verification technique, we focus on it because its role is often misunderstood, and as a result it doesn't get as much attention as other parts of the software lifecycle.

💡 Key Insight

Approaching software construction from a test-centric perspective often improves the software's readability and maintainability. In other words, testable code tends to be clear code, and vice versa. This insight may take a while to sink in if you are new to TDD — be patient and have faith in the process.

1.2 QA is a Shared Responsibility

In Agile development, developers do not “toss their code over the wall” to the Quality Assurance (QA) team, nor do QA engineers extensively exercise the software manually and file bug reports. Instead, Agile developers bear far more responsibility for testing their own code and participating in reviews, while Agile QA responsibilities focus on improving the testing tools infrastructure, helping developers make their code more testable, and verifying that customer-reported bugs are reproducible. Further- more, in the vast majority of tests you will write, the test code itself can determine whether the code being tested works or not, without requiring a human to manually check test output or interact with the software.
Even though Agile developers are expected to write their own tests, and those tests are expected to be automated, there is often a role for some manual testing:

  • User acceptance testing — observes actual users (or QA engineers acting as "typical" users) using the product to determine whether you "built the right thing"
  • Operational acceptance testing — manually tries additional scenarios to ensure you "built the thing right"

Both can uncover bugs that were previously undetected, some of which can then have automated tests created for them. Some visual aspects of design — such as whether particular elements render in a visually appealing way — require manual inspection. In general, modern software quality assurance is the shared responsibility of a whole team following good processes, rather than compartmentalized in a separate group.

1.3 Test-Driven Development (TDD)

TDD advocates the use of tests to drive the development of code. When TDD is used to create new code, as in this module, it is sometimes referred to as test-first development. The basic TDD workflow, repeated for each created test, is known as Red–Green–Refactor. Red–Green–Refactor proceeds as follows:

1 Write a failing test (think before you build)

Before you write any code, write a test for one aspect of the behavior you expect the new code will have. Since the code being tested doesn't exist yet, writing the test forces you to think about how you wish the code would behave and interact with its collaborators if it did exist. We call this "exercising the code you wish you had."

2 Red — run the test and watch it fail

Run the test and verify that it fails because you haven't yet implemented the code necessary to make it pass (that is, the code you wish you had).

3 Green — write the simplest code that passes

Write the simplest possible code that causes this test to pass without breaking any existing tests.

4 Refactor — clean up with confidence

Look for opportunities to refactor either your code or your tests — changing the code's structure to eliminate redundancy or repetition that may have arisen as a result of adding the new code. The tests ensure that your refactoring doesn't introduce bugs.

👀 How Do You Know When You're Done?

If you are using BDD (Module 2) to drive your application development, the new code being written is presumably necessary to make one or more Gherkin scenario steps pass. When all steps in a scenario pass, you're done.

Although TDD may feel strange at first, it tends to result in code that is not only well tested, but also more modular and easier to read than code developed separately from tests. While TDD is certainly not the only way to achieve those goals, it is difficult to end up with seriously deficient code if TDD is used correctly.

1.4 The FIRST Properties of a Good Test

What about the tests themselves? Five principles for creating good tests are summarized by the acronym FIRST: Fast, Independent, Repeatable, Self-checking, and Timely.

Independent

F Fast

It should be easy and quick to run the subset of test cases relevant to your current coding task, to avoid interfering with your train of thought.

I Independent

The order in which tests run shouldn't matter. More precisely, if no test relies on preconditions created by other tests, we can prioritize running only a subset of tests that cover recent code changes.

R Repeatable

Test behavior should not depend on external factors such as today's date or on "magic constants" that will break the tests if their values change — as occurred with many 1960s programs when the year 2000 arrived, due to the Y2K problem.

S Self-checking

Each test should be able to determine on its own whether it passed or failed, rather than relying on humans to check its output.

T Timely

Tests should be created or updated at the same time as the code being tested. As we'll see, with test-driven development the tests are written immediately before the code.

📝 Section Summary
  • Test-centric construction improves readability and maintainability: testable code tends to be clear code, and vice versa.
  • Software QA is a shared responsibility of the whole team: developers write most of their own tests, with QA staff improving the testing environment and handling kinds of testing that are hard to automate.
  • The TDD cycle of Red–Green–Refactor begins with writing a test that fails because the subject code doesn't exist yet (Red), then adding the minimum code necessary to pass just that one example (Green), and finally DRYing out and cleaning up the test code (Refactor).
  • Tests should be Fast, Independent of run order and thus Repeatable, Self-checking, and developed in a Timely way with respect to the code they test.

2. Anatomy of a Test Case: Arrange, Act, Assert

2.1 Key Vocabulary

Following the terminology in the widely-used xUnit Test Patterns (Meszaros 2007), we refer to the object being tested as the System Under Test (SUT) — whether that "object" is a single method, a group of methods, or even the entire application. That is, SUT is defined from the point of view of the test.

📋 Core Definitions
  • System Under Test (SUT) — the thing being tested, as small as a single method or as large as the whole app
  • Test suite — a collection of test cases. A codebase usually has several test suites, corresponding to different kinds of tests (we discuss this further in Section 6).
  • Unit test — the finest-grained test case, for which the SUT is a single method
  • Leaf method — a method that does not call any other methods to help do its job. Since even a leaf method may have multiple testable behaviors, a single method may be the subject of multiple test cases.
📋 The Structure of a Unit Test

In this section (section-2) we focus on unit tests. A unit test is conceptually simple: call a method, and verify some aspect of its behavior.
But even leaf methods usually require establishing some preconditions before exercising the code. For example, when testing a method that combines two lists into a single sorted list, we need to create the two lists that will be provided as input. We then exercise the SUT, and finally check whether the particular behavior we were looking for was correctly exhibited.

In general, every test case in a suite follows the same three-part structure:

1  ARRANGE

Create any necessary preconditions for the test case, such as setting values of variables that affect the behavior of the SUT.

2  ACT

Exercise the SUT — call the method or trigger the behavior you are testing.

3  ASSERT

Verify that the result or behavior matches what was expected.

2.2 Choosing Input Values: Critical Points and Categories

The only thing a test case needs to do is choose some inputs, call the method, and check the returned value. For example, consider a hypothetical method "leap" that accepts an integer and returns a truthy value if and only if that integer corresponds to a leap year (that is, it is either a multiple of 400, or a multiple of 4 but not 100). So, for example, 2000 and 2004 are leap years, but 1900 is not. Since exhaustive testing (trying every possible input) is clearly infeasible, how do we choose which input values to use for our test cases?

A good guideline in such cases is to use input values that would cause the calculation performed in the method to follow different code paths. Given the above rule for leap years, by inspection we can deduce four categories, as the following table shows:

# Category Test Value
1 A number that is not a multiple of 4 or 100 (and therefore not a multiple of 400) 1973
2 A number that is a multiple of 4, but not of 100 (and therefore not of 400) 2008
3 A number that is not a multiple of 400, but is a multiple of 100 1900
4 A number that is a multiple of 400 2000

The calculation of a leap year depends only on which category a value is in, not the value itself.

2.3 A "Pure" TDD Workflow for is_leap_year()

A "pure" TDD workflow for developing a function that detects leap years might proceed as follows:

  1. Choose any value in category 1 above, and write a test case that asserts that the return value of is_leap_year is falsy when called with that value.
  2. The test fails because is_leap_year doesn't yet exist, so write just enough of is_leap_year to make that case pass.
  3. Next, choose a value in category 2, write the corresponding test, and when it fails, modify is_leap_year so that now both tests pass.
  4. Continue until all categories are covered.
1 Category 1 — RED then GREEN: first test, first implementation
RED Write the test — function does not exist yet

Choose any value from category 1 (not a multiple of 4). Write a test asserting is_leap_year returns False for it. The function does not exist yet — so running pytest will immediately fail with a NameError. That is expected and correct.

test_leap_year.py — first test only 
def test_not_multiple_of_4():
    assert is_leap_year(1973) is False  # RED: is_leap_year doesn't exist yet

GREEN Write the minimum code to make it pass

Now create calendar_utils.py and write just enough code to make this one test pass — nothing more.

    assert is_leap_year(1973) is False  # still passing ✓

    assert is_leap_year(2008) is True   # GREEN: 2008 % 4 == 0 already passes ✓
✅ GREEN — both tests pass, no code change needed

The existing implementation already handles category 2. Both tests pass. Move on.

3 Category 3 — A new test exposes a bug (RED → fix → GREEN)

Category 3: a year divisible by 100 but not by 400 is not a leap year (e.g. 1900). Our current code returns True for 1900 because 1900 % 4 == 0. The new test catches this bug.

test_leap_year.py — three tests, one failing

    assert is_leap_year(1900) is False  # RED: current code wrongly returns True

Fix the implementation to handle the century-year exception:

calendar_utils.py — updated (GREEN) 
def is_leap_year(year):
    if year % 100 == 0:
        return False         # century years are NOT leap years...
    return year % 4 == 0  # ...unless divisible by 4
✅ GREEN — all three tests pass

1973 → False ✓  |  2008 → True ✓  |  1900 → False ✓

4 Category 4 — Final test reveals another edge case (RED → GREEN)

Category 4: a year divisible by 400 is a leap year (e.g. 2000). Our current code returns False for 2000 because 2000 % 100 == 0 and we return False for all century years. The new test catches this second edge case.

test_leap_year.py — four tests, one failing 
def test_multiple_of_400():
    assert is_leap_year(2000) is True   # RED: current code wrongly returns False

Fix the implementation one final time — now all four categories are handled:

📄 calendar_utils.py — final implementation (GREEN) 
def is_leap_year(year):
    if year % 400 == 0:
        return True          # divisible by 400 → always a leap year
    if year % 100 == 0:
        return False         # divisible by 100 but not 400 → not a leap year
    return year % 4 == 0  # divisible by 4 → leap year
✅ GREEN — all four tests pass

1973 → False ✓  |  2008 → True ✓  |  1900 → False ✓  |  2000 → True ✓

💡 "Why Not Just Write the Whole Method at Once?"

is_leap_year is such a simple leaf method that you might object that you could write the entire method at once along with all four test cases, rather than going through Red–Green–Refactor four separate times. For very simple leaf functions, this objection has merit — but notice what happened above: each new test exposed a bug in the previous implementation. The discipline of writing one test at a time scales to far more complex code, where it is not at all obvious what "the whole method" should look like in advance.

2.4 The Three Phases of a Test Case

Looking back at the four tests we just wrote, they all share the same structure. In each test we set up an input value, call the function, and check the result. This three-part pattern has a name: Arrange — Act — Assert.

📋 The Structure of Every Test Case

In general, every test case in a suite follows the same three-part structure:

1  ARRANGE

Create any necessary preconditions for the test case, such as setting values of variables that affect the behavior of the SUT.

2  ACT

Exercise the SUT — call the method or trigger the behavior you are testing.

3  ASSERT

Verify that the result or behavior matches what was expected.

The assertion step makes each test Self-checking, eliminating the need for a human programmer to inspect test results. Now that we have built up all four tests one at a time, here they are together — each one annotated to show which phase is which:

📄 test_leap_year.py — complete test suite 
from calendar_utils import is_leap_year

# Category 1: not a multiple of 4
def test_not_multiple_of_4():
    # Arrange: year = 1973 (the input value is the only setup needed)
    # Act + Assert combined — trivial arrange means both fit on one line
    assert is_leap_year(1973) is False

# Category 2: multiple of 4, not of 100
def test_multiple_of_4_not_100():
    assert is_leap_year(2008) is True

# Category 3: multiple of 100, not of 400
def test_multiple_of_100_not_400():
    assert is_leap_year(1900) is False

# Category 4: multiple of 400
def test_multiple_of_400():
    assert is_leap_year(2000) is True

In this simple case the Arrange phase is trivial — there is no setup needed beyond the input value itself, so Arrange, Act, and Assert collapse onto a single line. In more complex tests (covered in Section 3) the three phases are clearly separated. Common assertion patterns in pytest:

On a value Example (pytest)
Value equality assert x == "Python"  /  assert x != "Python"
Boolean (truthy) assert x (non-False / non-None)
Regular expression assert re.match(r"YourRegexHere", s)
Object type assert isinstance(a, list)
On a block / call Example (pytest)
Exception raised with pytest.raises(ValueError): math.sqrt(-1)
Side effect before = len(reviews); delete_first_review(); assert len(reviews) == before - 1

One can invert the sense of any assertion (assert x != 50), but as Module 2 warned, negative assertions should be used with caution — there are many ways for a program not to satisfy a particular condition while still not behaving correctly.

3. Isolating Code: Doubles and Seams

We can distinguish three characteristics — which may occur individually or together — that complicate unit testing:

  1. The SUT relies on results from calling collaborator methods or invoking behaviors on collaborator objects — like asking a friend for directions instead of knowing the route yourself: you don't control what they'll say back.
  2. The SUT has nondeterministic or time-dependent behavior — like asking "what's the weather?": the same question gets a different answer depending on when you ask.
  3. The SUT has side effects — it reaches outside itself and changes the world (writes to a database, sends an email) instead of just handing back a value.

3.1 A Motivating Example: Looking Up a Movie

Suppose a controller action tries to search a remote database for one or more matches to a movie title:

  • The Movie model has a function find_in_tmdb that calls the API of an external service, The Movie Database (TMDb), and returns a list of Movie objects, which may be empty if there were no matches.
  • If there are no matches, the controller action should redirect the user back to the search page with an appropriate message.
  • If there is exactly one match, the controller should render a view that allows the user to enter a review for that movie.
  • If there is more than one match, the controller should render a different view that allows the user to specify which movie they want to review.
  • Because the model method relies on calling an external service, the call might fail. In that case, we assume find_in_tmdb will raise an exception ConnectionError.
📄 views.py — review_movie 
def review_movie(request):
    search_string = request.GET.get("search")
    try:
        matches = Movie.find_in_tmdb(search_string)
        if not matches:  # nothing was found
            return redirect("review_movie", alert="No matches.")
        elif len(matches) == 1:
            movie = matches[0]
            return render(request, "review_movie.html", {"movie": movie})
        else:  # more than 1 match
            return render(request, "select_movie.html", {"movies": matches})
    except ConnectionError as err:
        return redirect("review_movie", alert=f"Error contacting TMDb: {err}")

Walking through it: find_in_tmdb() is the collaborator — the SUT (review_movie()) asks it to do the actual searching, then branches on whatever comes back:

1 find_in_tmdb() returns [ ] — zero matches

An empty list is "falsy," so not matches is True and the view redirects back to the search page with "No matches."

2 find_in_tmdb() returns a list of length 1 — one match

movie = matches[0] grabs that single result, and the view renders the review page so the user can write a review.

3 find_in_tmdb() returns a longer list — several matches

Neither condition above is true, so the else branch runs and the view renders a selection page asking "Which movie did you mean?"

4 find_in_tmdb() raises ConnectionError — the network fails

Control jumps straight to the except block, and the view redirects back to the search page with an error message instead of any matches.

How would we unit-test this view? The Arrange step consists of preparing a request with some search string. The Act step consists of calling the view with that search string. But the Assert step depends on whether the call to find_in_tmdb returns an empty list, a list of exactly one match, a list containing more than one match, or raises an exception because of an error communicating with TMDb. There are really four test cases required here, and to test each of them, we essentially need to be able to control the behavior of the call to find_in_tmdb.

3.2 Seams

🔑 Definition: Seam

Michael Feathers (2004) defines a seam as "a place where you can alter behavior in your program without editing in that place." In our case, we want to alter (control) the behavior of find_in_tmdb without changing the source code of the view function. Think of it like a light socket: you swap the bulb, the switch never changes.

The strategy is to temporarily replace find_in_tmdb so that instead of calling the real method, it calls a "fake" method whose behavior we control and can change for each test case. Such a construction is called a method stub. In pytest, this is most commonly done using the monkeypatch fixture or the unittest.mock library.

📄 test_views.py 
def test_redirects_to_search_if_no_match(client, monkeypatch):
    # Arrange: stub Movie.find_in_tmdb to return an empty list
    monkeypatch.setattr(Movie, "find_in_tmdb", lambda q: [])

    # Act
    response = client.get("/review_movie", {"search": "I Am Big Bird"})

    # Assert
    assert response.status_code == 302
    assert response.url == reverse("review_movie")

In plain terms, that monkeypatch.setattr line says: "inside Movie, replace find_in_tmdb with lambda q: []" — a function that ignores its argument and always returns an empty list. This is the Arrange part of the test: setting up a stub for the method, specifying that when it's called, it should return an empty list — ensuring that not matches is true in our view, causing the redirect branch to execute next.

As is typical for a testing framework, monkeypatch automatically undoes the patch after each test function finishes, making the stub visible only to that test case and thereby keeping tests Independent. Note that it is not an error for find_in_tmdb to never be called — the stub setup only specifies what should happen if that method is called. If we wanted to express the test condition that the method must be called, we would use a Mock object and assert that it was called, as shown next.

3.3 Stubs vs. Mocks vs. Spies

📐 Test Doubles — A Vocabulary
  • Stub — a piece of code that replaces the real method's code with a controllable or fixed behavior for testing purposes. A stub only specifies what should happen if it's called.
  • Mock — a simplified "stunt double" of an object that can only mimic a few fixed behaviors, such as returning fixed values for specific attributes. Mocks are useful when a real object would be complex to instantiate because it has other dependencies, yet only a few specific properties of the object are necessary for the SUT to work properly. A mock additionally lets you assert that it was called.
  • Spy — similar to a stub, but lets a call to the real method proceed while "recording" the arguments and return values for later inspection.
  • Test double — the generic term covering all of the above and a few other types of seams.
  Runs the real code? Controls the return value? Lets you check it was called?
Stub No Yes No
Mock No Yes Yes
Spy Yes — Yes

The term test double generically covers stubs, mocks, and spies. Figure 8.6 in the textbook summarizes typical strategies for using doubles in various unit-testing scenarios; the table below shows the same strategies translated to pytest.

System Under Test (SUT) Testing Strategy
Pure leaf function — no side effects, no collaborator methods or classes, same inputs yield same outputs Assert correct output results for critical values and for arbitrary values in noncritical regions
Relies on results from calling collaborator methods or invoking behaviors on collaborator objects In Arrange phase, create doubles that "force" the desired behavior by returning prearranged values, raising an exception, and so on
Nondeterministic or time-dependent behavior In code under test, isolate the nondeterminism in a method call that can be stubbed using a double in the Arrange phase
Has side effects In Arrange phase, observe the relevant state before executing test code; in Assert phase, observe it again and check for the side effect

Here are the four strategies above illustrated with concrete, everyday examples from a school grading app — a domain every student already understands. For each case the code under test is shown first so you can see exactly what is being tested, then the test itself:

📝 Case 1 — Pure leaf function: just call it and check

letter_grade(score) does pure arithmetic. It calls nothing else, so there is nothing to fake.

Code under test — grades.py
def letter_grade(score):
    if score >= 90: return "A"
    if score >= 80: return "B"
    if score >= 70: return "C"
    if score >= 60: return "D"
    return "F"
Test — test_grades.py
# Arrange: nothing needed — the input IS the setup
# Act + Assert in one line: call the function, check the result
def test_grade_for_90_is_A():
    assert letter_grade(90) == "A"

def test_grade_for_60_is_D():
    assert letter_grade(60) == "D"
☁️ Case 2 — Collaborator dependency: fake the thing you don't control

get_status_message() depends on weather.is_raining(), which calls a real weather API. We can't control whether it actually rains when the test runs, so we fake is_raining() to return exactly what we need.

Code under test — status.py
def get_status_message():
    if weather.is_raining():   # <-- calls external weather API
        return "Don't forget your umbrella!"
    return "Have a nice day!"
Test — test_status.py
def test_umbrella_reminder_when_raining(monkeypatch):
    # Arrange: replace weather.is_raining with a fake that always returns True
    # (we don't want the test to depend on actual weather!)
    monkeypatch.setattr(weather, "is_raining", lambda: True)

    # Act + Assert: call the function, check it picks the rainy message
    assert get_status_message() == "Don't forget your umbrella!"
📅 Case 3 — Implicit time dependency: freeze the clock

get_discount_percent() gives 50% off on Black Friday. If you run this test on any other day of the year, it would return 0 and the test would fail — even though the code is correct. So we freeze the date to Black Friday so the test always behaves the same way regardless of when it's run.

Code under test — shop.py
from datetime import date

def get_discount_percent():
    if date.today() == date(2024, 11, 29):  # Black Friday 2024
        return 50
    return 0
Test — test_shop.py
from datetime import date

def test_50_percent_discount_on_black_friday(monkeypatch):
    # Arrange: freeze date.today() to Black Friday — test always passes,
    # regardless of what day it actually is when you run the test
    black_friday = date(2024, 11, 29)
    monkeypatch.setattr(date, "today", classmethod(lambda cls: black_friday))

    # Act + Assert
    assert get_discount_percent() == 50
🏫 Case 4 — Side effects: check state before and after

enroll_student("Alice") doesn't return anything useful — its whole job is to change the classroom's state by adding a student to the roster. To test this, we record the class size before the call and verify it grew by exactly 1 afterward.

Code under test — classroom.py
roster = []

def student_count():
    return len(roster)

def enroll_student(name):
    roster.append(name)  # no return value — changes state as a side effect
Test — test_classroom.py
def test_enroll_increases_class_size_by_1():
    # Arrange: snapshot the count BEFORE the action
    before = student_count()

    enroll_student("Alice")  # Act: do the thing

    # Assert: the count AFTER must be exactly one more than before
    assert student_count() == before + 1
Case What makes it tricky Solution
1. letter_grade() — leaf function Nothing — it's entirely self-contained Just call it and check the return value
2. get_status_message() — collaborator Depends on weather.is_raining(), which changes every day Fake is_raining() with monkeypatch to return a fixed value
3. get_discount_percent() — time dependency Result depends on today's date, which changes every day Freeze date.today() to a known date with monkeypatch
4. enroll_student() — side effect Returns nothing — only changes state Record state before, call the function, compare state after

Asserting that a Mock Was Called

If we want to express the test condition that a method must be called — not just stub its return value — we use unittest.mock.Mock or MagicMock and assert on the call afterward:

from unittest.mock import patch

def test_find_in_tmdb_is_called_with_search_string():
    with patch.object(Movie, "find_in_tmdb", return_value=[]) as mock_find:
        client.get("/review_movie", {"search": "Milk"})

    # Assert: the stub MUST have been called with "Milk"
    mock_find.assert_called_once_with("Milk")
📝 Section Summary
  • When testing a method that has external dependencies — calling other methods or consuming other objects — we use test doubles to "stand in" for the real methods or objects and allow the test to tightly control the SUT's behavior.
  • Test doubles are set up in the Arrange phase of a test case. Stubs are set up to control return values from collaborator methods, while mocks are set up to mimic just those behaviors of the collaborator object used by the SUT.
  • Depending on what behavior is being tested, a test case can specify whether a particular stub must be called — that is, if the stub not being called signals a bug.
❓ Self-Check 8.3.1

Name two likely violations of FIRST that arise when unit tests actually call an external service as part of testing.

The test may no longer be Fast, since it takes much longer to call an external service than to compute locally. The test may no longer be Repeatable, since circumstances beyond our control could affect its outcome, such as the temporary unavailability of the external service.

4. Stubbing the Internet

When testing code that calls an external API, we almost never want to make a real call to that API — it's slow, unreliable, and can even get you in trouble.

⚠️ A Real Cautionary Tale

A CS department once received a complaint after students' "test" runs repeatedly hit a real external API — the site threatened to cut off the university's access.

Real API calls also threaten Repeatable and Fast test cases. All we really care about when testing our own app is whether the calls we'd make are correctly formed — so the real question is: where should we stub the external service?

4.1 Stubbing "Close By" vs. "Far Away"

In Section 3 we stubbed the Movie model itself — "close by." A more realistic alternative is to stub "farther away," intercepting the actual HTTP call and returning canned JSON, using fixtures that mimic real responses from the service.

🌐 Python Tooling

responses stubs out the web except for specific URIs, which return a canned response. VCR.py automates this further: it records a real response once into a fixture file (a cassette) and replays it on later test runs.

📄 test_tmdb_integration.py — stubbing "far away" with responses 
import responses

@responses.activate
def test_find_in_tmdb_returns_one_match():
    # Arrange: intercept the TMDb URL and return canned JSON
    responses.add(
        responses.GET,
        "https://api.themoviedb.org/3/search/movie",
        json={"results": [{"title": "Milk", "id": 7508}]},
        status=200,
    )

    # Act
    matches = Movie.find_in_tmdb("Milk")

    # Assert
    assert len(matches) == 1
    assert matches[0].title == "Milk"

responses is more realistic for integration tests (e.g. Behave/pytest-bdd scenarios), since it stubs as late as possible. For pure unit tests, stubbing "close by" (Section 3) is usually enough.

📝 Section Summary
  • Stub external services to keep tests Fast and Repeatable.
  • Stub "far away" (e.g. with responses) for integration tests; stub "close by" for unit tests.
❓ Self-Check 8.4.1

Is "stubbing the Internet" in conflict with the advice of Module 2 that one should avoid mocks or stubs in full-system Behave/pytest-bdd scenarios?

Full-system testing should avoid faking most of the application, but if the "full system" includes an outside service you don't control — like TMDb here — you still need to fake that to test reliably.

5. Fixtures and Factories

Section 3 was about doubles — fake stand-ins for a collaborator you don't control, like the TMDb API. But now imagine testing Movie.name_with_rating(), a method that just formats a movie's own title and rating into a string like "Milk (R)".

💡 There's No Collaborator Here

This method isn't asking anyone else for help — it's just working with its own object's data. Faking anything would be pointless; you'd just use a real Movie object. The real question becomes: what's the fastest way to get a real, valid Movie into a test? That's what factories and fixtures both answer, in two different ways.

5.1 Factories: A "Fill-in-the-Blanks" Template for Objects

Imagine a paper form for registering a new movie in a database. It has fields for title, rating, release date, director, runtime, and so on — most of them required, or the form won't submit. But for a given test, you might only care about one or two of those fields (say, title and rating).

A factory is a template with that form pre-filled with sensible defaults for every field. When you write a test, you say "give me a movie, but override just the title and rating" — the factory fills in everything else automatically so the object is valid. In Python, the factory_boy library does this.

📄 tests/factories.py & test_movie.py 
class MovieFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Movie

    title = "A Fake Title"   # the form's pre-filled defaults
    rating = "PG"


def test_name_with_rating_includes_title_and_rating():
    # "Give me a movie, but override just title and rating"
    movie = MovieFactory.build(title="Milk", rating="R")
    assert movie.name_with_rating() == "Milk (R)"
🔍 Reading That Code

class MovieFactory(factory.django.DjangoModelFactory) just means "this blueprint borrows its powers from a ready-made tool that knows how to build database objects." class Meta: model = Movie tells it which object to build. title and rating below that are the pre-filled defaults — used unless a test says otherwise.

In the test, MovieFactory.build(title="Milk", rating="R") stamps out a real Movie, overriding just those two fields; anything else the model needed would still fall back to the factory's defaults. The result, movie, is an ordinary Movie object — the factory just saved you from typing every field by hand.

build() creates the object without saving it to the database; create() also saves it. (Don't confuse this "factory" with the Abstract Factory design pattern from Module 6 — same name, unrelated idea.)

Factories become genuinely important — not just a convenience — in two situations:

  1. Objects with lots of required fields you don't want to type out by hand every single time.
  2. Objects with relationships — e.g. a Review belongs to a Movie, so you literally can't create a valid Review without a Movie to go with it. The factory can be told "when you make a Review, automatically make a Movie for it to belong to."

5.2 Fixtures: Objects That Already Exist, Waiting For You

A fixture takes a different approach: instead of building a fresh object every time a test needs one, you pre-load a known object into the test database once, and every test that needs it just uses it — like a prop that's already sitting on set before filming starts, rather than someone building it fresh for each scene.

📄 conftest.py & test_movie.py 
@pytest.fixture
def hamilton(db):
    # a real Movie, already saved to the test database
    return MovieFactory.create(title="Hamilton", rating="PG-13")


def test_hamilton_has_correct_rating(hamilton):
    # the test just asks for "hamilton" by name and pytest hands it over
    assert hamilton.rating == "PG-13"

Side note: pytest also uses the word "fixture" more loosely for any reusable setup — a database connection, a temp file, some test data — not just database objects. Same word, slightly broader meaning.

5.3 Why Factories Are Usually Preferred

⚠️ The Catch With Fixtures

If ten different tests all use the same hamilton fixture, they're all secretly relying on the same shared object. If one test changes something about it, or you tweak the fixture's defaults, you might accidentally break other tests that assumed something else about hamilton. That breaks Independence — each test is supposed to stand on its own.

Factories don't have this problem, because every test builds its own fresh object on the spot. Nothing is shared, so nothing can leak between tests.

📝 The One-Sentence Version
  • Testing a method that talks to a collaborator? → use a double (Section 3).
  • Testing a method that just needs a real, valid object of its own class? → build one with a factory — fast, safe, no shared state.
  • Only reach for a fixture when you truly need one fixed object sitting in the database ahead of time — and even then, sparingly, since it creates shared state that tests can accidentally trip over.

6. Coverage Concepts and Types of Tests

6.1 How Much Testing Is Enough?

"As much as you can before the deadline" is a bad answer. A better one is code coverage: a measurement of how thoroughly your test suite actually exercises your code.

💡 The Intuition: Coverage = Exploring a Maze

Think of your code's logic as a maze of branching paths. Every if, else, and condition forks the maze further. A test "walks" one path through it. Coverage asks how much of the maze your test suite has actually walked — not just whether you wrote a lot of tests.

Consider this small class, used throughout this section:

📄 my_class.py 
class MyClass:
    def foo(self, x, y, z):
        if x:
            if y and z:
                self.bar(0)
        else:
            self.bar(1)

    def bar(self, x):
        self.w = x

6.2 The Coverage Ladder

Each level below asks a stricter question about how thoroughly the maze was explored:

Level Plain-language question For MyClass, satisfied by…
S0 Did we call every method at all? Call foo and bar once each
S1 Did we call each method from every place that calls it? Call bar from both inside and outside foo's if
C0 Did we run every line at least once? Run foo once with x true, once with y false
C1 Did every branch go both ways at least once? Try x both True/False, and y and z both True/False
C2 Did we walk every possible route through the code? All 8 combinations of x, y, z

An even stricter standard, MC/DC (required by the FAA for flight software), demands proof that each individual condition — not just the overall branch — independently changes the outcome. It's the gold standard for safety-critical code, not typical SaaS work.

C0 is realistic to push toward 100%. C1 takes real care to construct test cases for. C2 explodes combinatorially and isn't even universally agreed to be worth chasing.

💡 The Real Point of Coverage Numbers

Coverage is most useful for pointing at what's untested, not as a score to maximize. 100% coverage doesn't mean your tests are good — it just means every line ran at least once under some assertion, or none at all.

In Python, run coverage with pip install pytest-cov then pytest --cov=myapp --cov-report=term-missing — it reports which exact lines your tests never touched.

6.3 Unit vs. Functional vs. System Tests

This is the classic testing pyramid tradeoff: the smaller and more isolated a test is, the faster it runs and the more precisely it points at what broke — but the less it proves about the app as a real user would experience it.

Unit Functional System / Integration
Scope One method/class Several methods/classes Large chunks of the app
Speed Very fast Fairly fast Slow
Error localization Excellent Moderate Poor
Uses doubles Frequently Occasionally Rarely

Neither end is "better" — unit tests catch bugs fast and pinpoint them precisely, but their heavy use of doubles can hide problems that only show up when real pieces interact. System/integration tests catch those problems, but slowly and vaguely. You need a mix of all three.

6.4 Beyond Correctness

A well-rounded suite also includes: smoke tests (a fast sanity check — is anything obviously broken before we bother running the rest?), compatibility tests (does it work across browsers/OSes?), regression tests (do old bugs stay fixed?), performance/ stress/security tests (does it hold up under real-world conditions?), and accessibility tests (can everyone actually use it?).

📝 Section Summary
  • Coverage (S0/S1/C0/C1/C2) measures how much of the code's "maze" your tests actually walk — use it to find gaps, not as a score to maximize.
  • Unit tests are fast and precise but narrow; system/integration tests are slow and blunt but realistic. High assurance needs both.
  • Correctness isn't the whole picture — smoke, compatibility, regression, performance, and accessibility testing all matter too.
❓ Self-Check

Why does high test coverage not necessarily imply a well-tested application?

Coverage says nothing about whether the assertions are any good — a line can "run" under a test that checks nothing meaningful. Low coverage, however, does reliably mean the app is poorly tested.

7. Other Testing Approaches and Terminology

💡 The One Idea Behind Everything In This Section

Every technique below is really the same move applied to a different "space": picture your code's possible behaviors as a graph you could explore, and ask how much of it did our tests actually visit? Section 6's coverage levels covered control flow (which lines/branches ran). The techniques here apply that same question to variables, inputs, and bugs themselves.

7.1 Define–Use (DU) Coverage — "Did we test every variable's story?"

Every time a variable is assigned a value, that's a define. Every time it's later read, that's a use. DU-coverage asks: has every define→use path been tested?

discount = 0 # define
if is_member:
    discount = 10 # define
apply_discount(discount) # use

A test suite that only ever runs with is_member = True never actually exercises the "discount stays 0" path — it has incomplete DU-coverage, and could easily be hiding a bug in that default.

7.2 Black-Box vs. White-Box — "What do you allow yourself to know?"

Black-Box White-Box
Test design uses only the spec — treat the code as a sealed box Test design uses knowledge of the implementation
e.g. store/read key-value pairs in a hash table, check you get back what you stored e.g. craft inputs known to cause hash collisions, since you can see the hash function

7.3 Mutation Testing — "Would our tests even notice a bug?"

A tool deliberately breaks your code in small ways (a mutation, e.g. flipping >= to >) and reruns your test suite. If every test still passes, your tests didn't actually notice the bug — a gap that plain coverage would never reveal, since the line still "ran." Python's mutmut automates this.

7.4 Fuzz Testing — "What if we just throw garbage at it?"

Feed the app huge volumes of random or semi-random data and see what breaks. Google reported fuzzing found 1,000+ bugs in ffmpeg over two years — especially good at finding security holes that manual review misses.

  • Random fuzzing — pure noise, or small random tweaks to valid input.
  • Smart fuzzing — noise shaped by knowledge of the app (e.g. trying SQL-injection-style payloads in form fields).
  • White-box fuzzing — simulates execution to specifically target branches never taken; can in principle reach C2 coverage, but the search space is huge in practice.

Python's hypothesis library brings this into pytest: it generates many random inputs and checks that a property you specify always holds.

📝 Section Summary

All of these are the same idea — "cover a graph of possible behaviors" — pointed at different targets: DU-coverage at variables, mutation testing at whether tests catch specific bugs, and fuzz testing at the space of possible inputs. They're complementary, each catching bugs the others miss.

❓ Self-Check

The Zune music player famously "locked up" every leap year on Dec 31. Which kind of testing — black-box, white-box, mutation, or fuzz — would most likely have caught it?

A white-box test targeting the leap-year code path directly, or fuzz testing (the bug recurs roughly every 1,460 days, so a few thousand random date inputs would likely have hit it).

8. The Plan-and-Document Perspective on Testing

💡 Zooming Out

Sections 1–7 covered how to write good tests. This section is about where testing fits in a traditional, big-process ("plan-and-document" / waterfall-style) project — as opposed to the Agile/TDD approach used elsewhere in this course, where tests are written before the code.

In this world, a project manager splits the spec into units. Developers write the code, then unit-test it. QA staff then handle the higher-level tests — module, integration, system, and acceptance.

8.1 Integration Strategies

Once units are unit-tested individually, how do you assemble and test them together? Think of it like building a house: do you build the roof first (propped up on temporary supports) so stakeholders can see the shape early, or build the foundation up solidly but not see the whole picture until near the end?

1 Top-down — start from the UI, work down

Shows stakeholders working high-level features fast, but needs lots of stubs to fake the not-yet-built lower layers.

2 Bottom-up — start from the foundation, work up

No stubs needed, but nobody sees how the app looks or feels until nearly everything is integrated.

3 Sandwich — both directions at once

A middle ground: integrate the highest-value pieces top-down and bottom-up simultaneously to balance stub cost against early feedback.

8.2 System Test and Validation

After integration comes the system test — the full app, checked against both functional and non-functional requirements, right before customers see it. This is about validation ("did we build the right thing?"), not just verification ("did we build the thing right?"). Agile skips a separate system-test phase because customers already see working prototypes throughout development.

Testing is usually declared "complete" once the team hits a pre-agreed coverage bar (e.g. every statement executed at least once, every input field tried with both good and bad data) — not some proof that all bugs are gone.

8.3 Formal Methods: Proving Instead of Testing

Since testing can never prove bugs are absent, formal methods try to mathematically prove code matches a formal spec:

  • Automatic theorem proving — builds a step-by-step logical proof of correctness, the way a mathematician proves a theorem, instead of just running the code and hoping.
  • Model checking — exhaustively tries every possible system state (e.g. every combination of a traffic light's colors) to prove a bad state, like "green both ways," can never happen.

Both are powerful but expensive — the state space explodes for large systems, so formal methods are mainly used for small, critical components (hardware, network protocols), not whole apps.

8.4 Test Plans and Plan-and-Document vs. Agile

Plan-and-document processes formalize testing into paperwork — a test plan (per IEEE Std 829-2008) documenting scope, test levels, responsibilities, and reporting requirements, in contrast to Agile's lightweight user stories. The table below summarizes the bigger-picture differences:

Task Plan-and-Document Agile
Documentation Formal test docs (IEEE 829) User stories
Order Code → unit → module → integration → system → acceptance Acceptance → integration → module → unit → code
Who tests Devs (unit); QA (everything higher-level) Developers, throughout
Testing stops when… Company policy / coverage bar is met All tests pass (green)
📝 Section Summary
  • Plan-and-document writes code before tests (opposite of TDD), with QA handling the higher-level test levels.
  • Units are integrated top-down, bottom-up, or sandwich — trading off early stakeholder feedback against stub overhead.
  • A final system test checks the whole app before customers validate it.
  • Testing can only show bugs exist, never that they don't — formal methods can mathematically prove correctness, but are too expensive for anything but small, critical components.

9. Fallacies and Pitfalls

🚧 Fallacy: 100% Test Coverage with All Tests Passing Means No Bugs

There are many reasons this statement can be false. Complete test coverage says nothing about the quality of the individual tests. As well, some bugs may require passing a certain value as a method argument (for example, to trigger a divide-by-zero error), and control flow testing often cannot reveal such a bug. There may be bugs in the interaction between your app and an external service such as TMDb; stubbing out the service so you can perform local testing might mask such bugs.

⚠️ Pitfall: Dogmatically Insisting on 100% Test Coverage (All Green) Before You Ship

As we saw above, 100% test coverage is not only difficult to achieve at levels higher than C1, but gives no guarantees of bug-freedom even if you do achieve it. Test coverage is a useful tool for estimating the overall comprehensiveness of your test suite, but high confidence requires a variety of testing methods — integration as well as unit, fuzzing as well as hand-constructed test cases, define–use coverage as well as control-flow coverage, mutation testing to expose additional holes in the test strategy, and so on. Operational issues such as security and performance call for additional testing strategies beyond the correctness-oriented ones described in this module.

🚧 Fallacy: You Don't Need Much Test Code to Be Confident in the Application

While insisting on 100% coverage may be counterproductive, so is going to the other extreme. In production systems, the code-to-test ratio is usually less than 1 — that is, there are typically more lines of test code than lines of application code. A small test suite, even one with high statement coverage, is unlikely to exercise the variety of inputs, edge cases, and error conditions that a real application encounters in production.
