1. FIRST, TDD, and Red–Green–Refactor

1.1 From Validation to Verification

Module 2 distinguished two aspects of software assurance:

Validation Verification
"Did you build the right thing?" "Did you build the thing right?"

This module focuses on verification — building the thing right — via software testing as part of the Agile lifecycle. Although testing is only one verification technique, we focus on it because its role is often misunderstood, and as a result it doesn't get as much attention as other parts of the software lifecycle.

💡 Key Insight

Approaching software construction from a test-centric perspective often improves the software's readability and maintainability. In other words, testable code tends to be clear code, and vice versa. This insight may take a while to sink in if you are new to TDD — be patient and have faith in the process.

1.2 QA is a Shared Responsibility

In Agile development, developers do not “toss their code over the wall” to the Quality Assurance (QA) team, nor do QA engineers extensively exercise the software manually and file bug reports. Instead, Agile developers bear far more responsibility for testing their own code and participating in reviews, while Agile QA responsibilities focus on improving the testing tools infrastructure, helping developers make their code more testable, and verifying that customer-reported bugs are reproducible. Further- more, in the vast majority of tests you will write, the test code itself can determine whether the code being tested works or not, without requiring a human to manually check test output or interact with the software.
Even though Agile developers are expected to write their own tests, and those tests are expected to be automated, there is often a role for some manual testing:

  • User acceptance testing — observes actual users (or QA engineers acting as "typical" users) using the product to determine whether you "built the right thing"
  • Operational acceptance testing — manually tries additional scenarios to ensure you "built the thing right"

Both can uncover bugs that were previously undetected, some of which can then have automated tests created for them. Some visual aspects of design — such as whether particular elements render in a visually appealing way — require manual inspection. In general, modern software quality assurance is the shared responsibility of a whole team following good processes, rather than compartmentalized in a separate group.

1.3 Test-Driven Development (TDD)

TDD advocates the use of tests to drive the development of code. When TDD is used to create new code, as in this module, it is sometimes referred to as test-first development. The basic TDD workflow, repeated for each created test, is known as Red–Green–Refactor. Red–Green–Refactor proceeds as follows:

1 Write a failing test (think before you build)

Before you write any code, write a test for one aspect of the behavior you expect the new code will have. Since the code being tested doesn't exist yet, writing the test forces you to think about how you wish the code would behave and interact with its collaborators if it did exist. We call this "exercising the code you wish you had."

2 Red — run the test and watch it fail

Run the test and verify that it fails because you haven't yet implemented the code necessary to make it pass (that is, the code you wish you had).

3 Green — write the simplest code that passes

Write the simplest possible code that causes this test to pass without breaking any existing tests.

4 Refactor — clean up with confidence

Look for opportunities to refactor either your code or your tests — changing the code's structure to eliminate redundancy or repetition that may have arisen as a result of adding the new code. The tests ensure that your refactoring doesn't introduce bugs.

👀 How Do You Know When You're Done?

If you are using BDD (Module 2) to drive your application development, the new code being written is presumably necessary to make one or more Gherkin scenario steps pass. When all steps in a scenario pass, you're done.

Although TDD may feel strange at first, it tends to result in code that is not only well tested, but also more modular and easier to read than code developed separately from tests. While TDD is certainly not the only way to achieve those goals, it is difficult to end up with seriously deficient code if TDD is used correctly.

1.4 The FIRST Properties of a Good Test

What about the tests themselves? Five principles for creating good tests are summarized by the acronym FIRST: Fast, Independent, Repeatable, Self-checking, and Timely.

F
Fast
I
Independent
R
Repeatable
S
Self-checking
T
Timely
F Fast

It should be easy and quick to run the subset of test cases relevant to your current coding task, to avoid interfering with your train of thought.

I Independent

The order in which tests run shouldn't matter. More precisely, if no test relies on preconditions created by other tests, we can prioritize running only a subset of tests that cover recent code changes.

R Repeatable

Test behavior should not depend on external factors such as today's date or on "magic constants" that will break the tests if their values change — as occurred with many 1960s programs when the year 2000 arrived, due to the Y2K problem.

S Self-checking

Each test should be able to determine on its own whether it passed or failed, rather than relying on humans to check its output.

T Timely

Tests should be created or updated at the same time as the code being tested. As we'll see, with test-driven development the tests are written immediately before the code.

📝 Section Summary
  • Test-centric construction improves readability and maintainability: testable code tends to be clear code, and vice versa.
  • Software QA is a shared responsibility of the whole team: developers write most of their own tests, with QA staff improving the testing environment and handling kinds of testing that are hard to automate.
  • The TDD cycle of Red–Green–Refactor begins with writing a test that fails because the subject code doesn't exist yet (Red), then adding the minimum code necessary to pass just that one example (Green), and finally DRYing out and cleaning up the test code (Refactor).
  • Tests should be Fast, Independent of run order and thus Repeatable, Self-checking, and developed in a Timely way with respect to the code they test.

2. Anatomy of a Test Case: Arrange, Act, Assert

2.1 Key Vocabulary

Following the terminology in the widely-used xUnit Test Patterns (Meszaros 2007), we refer to the object being tested as the System Under Test (SUT) — whether that "object" is a single method, a group of methods, or even the entire application. That is, SUT is defined from the point of view of the test.

📋 Core Definitions
  • System Under Test (SUT) — the thing being tested, as small as a single method or as large as the whole app
  • Test suite — a collection of test cases. A codebase usually has several test suites, corresponding to different kinds of tests (we discuss this further in Section 6).
  • Unit test — the finest-grained test case, for which the SUT is a single method
  • Leaf method — a method that does not call any other methods to help do its job. Since even a leaf method may have multiple testable behaviors, a single method may be the subject of multiple test cases.
📋 The Structure of a Unit Test

In this section (section-2) we focus on unit tests. A unit test is conceptually simple: call a method, and verify some aspect of its behavior.
But even leaf methods usually require establishing some preconditions before exercising the code. For example, when testing a method that combines two lists into a single sorted list, we need to create the two lists that will be provided as input. We then exercise the SUT, and finally check whether the particular behavior we were looking for was correctly exhibited.

In general, every test case in a suite follows the same three-part structure:

1  ARRANGE

Create any necessary preconditions for the test case, such as setting values of variables that affect the behavior of the SUT.

2  ACT

Exercise the SUT — call the method or trigger the behavior you are testing.

3  ASSERT

Verify that the result or behavior matches what was expected.

2.2 Choosing Input Values: Critical Points and Categories

The only thing a test case needs to do is choose some inputs, call the method, and check the returned value. For example, consider a hypothetical method "leap" that accepts an integer and returns a truthy value if and only if that integer corresponds to a leap year (that is, it is either a multiple of 400, or a multiple of 4 but not 100). So, for example, 2000 and 2004 are leap years, but 1900 is not. Since exhaustive testing (trying every possible input) is clearly infeasible, how do we choose which input values to use for our test cases?

A good guideline in such cases is to use input values that would cause the calculation performed in the method to follow different code paths. Given the above rule for leap years, by inspection we can deduce four categories, as the following table shows:

# Category Test Value
1 A number that is not a multiple of 4 or 100 (and therefore not a multiple of 400) 1973
2 A number that is a multiple of 4, but not of 100 (and therefore not of 400) 2008
3 A number that is not a multiple of 400, but is a multiple of 100 1900
4 A number that is a multiple of 400 2000

The calculation of a leap year depends only on which category a value is in, not the value itself.

2.3 A "Pure" TDD Workflow for is_leap_year()

A "pure" TDD workflow for developing a function that detects leap years might proceed as follows:

  1. Choose any value in category 1 above, and write a test case that asserts that the return value of is_leap_year is falsy when called with that value.
  2. The test fails because is_leap_year doesn't yet exist, so write just enough of is_leap_year to make that case pass.
  3. Next, choose a value in category 2, write the corresponding test, and when it fails, modify is_leap_year so that now both tests pass.
  4. Continue until all categories are covered.
1 Category 1 — RED then GREEN: first test, first implementation
RED Write the test — function does not exist yet

Choose any value from category 1 (not a multiple of 4). Write a test asserting is_leap_year returns False for it. The function does not exist yet — so running pytest will immediately fail with a NameError. That is expected and correct.

test_leap_year.py — first test only 
def test_not_multiple_of_4():
    assert is_leap_year(1973) is False  # RED: is_leap_year doesn't exist yet
❌ Test output at this point

ERROR: NameError: name 'is_leap_year' is not defined

GREEN Write the minimum code to make it pass

Now create calendar_utils.py and write just enough code to make this one test pass — nothing more.

calendar_utils.py — bare minimum 
def is_leap_year(year):
    return year % 4 == 0  # just enough for category 1 — nothing more yet
✅ Test output now

test_not_multiple_of_4  PASSED  ✔   —   is_leap_year(1973) returns False because 1973 % 4 ≠ 0.

2 Category 2 — Add a test, watch it fail, fix the code (RED → GREEN)

Choose a value from category 2 (multiple of 4, not of 100). Add a second test. The current code already handles this correctly — but add the test anyway to lock in this behavior.

test_leap_year.py — two tests now 
def test_not_multiple_of_4():
    assert is_leap_year(1973) is False  # still passing ✓

def test_multiple_of_4_not_100():
    assert is_leap_year(2008) is True   # GREEN: 2008 % 4 == 0 already passes ✓
✅ GREEN — both tests pass, no code change needed

The existing implementation already handles category 2. Both tests pass. Move on.

3 Category 3 — A new test exposes a bug (RED → fix → GREEN)

Category 3: a year divisible by 100 but not by 400 is not a leap year (e.g. 1900). Our current code returns True for 1900 because 1900 % 4 == 0. The new test catches this bug.

test_leap_year.py — three tests, one failing 
def test_multiple_of_100_not_400():
    assert is_leap_year(1900) is False  # RED: current code wrongly returns True

Fix the implementation to handle the century-year exception:

calendar_utils.py — updated (GREEN) 
def is_leap_year(year):
    if year % 100 == 0:
        return False         # century years are NOT leap years...
    return year % 4 == 0  # ...unless divisible by 4
✅ GREEN — all three tests pass

1973 → False ✓  |  2008 → True ✓  |  1900 → False ✓

4 Category 4 — Final test reveals another edge case (RED → GREEN)

Category 4: a year divisible by 400 is a leap year (e.g. 2000). Our current code returns False for 2000 because 2000 % 100 == 0 and we return False for all century years. The new test catches this second edge case.

test_leap_year.py — four tests, one failing 
def test_multiple_of_400():
    assert is_leap_year(2000) is True   # RED: current code wrongly returns False

Fix the implementation one final time — now all four categories are handled:

📄 calendar_utils.py — final implementation (GREEN) 
def is_leap_year(year):
    if year % 400 == 0:
        return True          # divisible by 400 → always a leap year
    if year % 100 == 0:
        return False         # divisible by 100 but not 400 → not a leap year
    return year % 4 == 0  # divisible by 4 → leap year
✅ GREEN — all four tests pass

1973 → False ✓  |  2008 → True ✓  |  1900 → False ✓  |  2000 → True ✓

💡 "Why Not Just Write the Whole Method at Once?"

is_leap_year is such a simple leaf method that you might object that you could write the entire method at once along with all four test cases, rather than going through Red–Green–Refactor four separate times. For very simple leaf functions, this objection has merit — but notice what happened above: each new test exposed a bug in the previous implementation. The discipline of writing one test at a time scales to far more complex code, where it is not at all obvious what "the whole method" should look like in advance.

2.4 The Three Phases of a Test Case

Looking back at the four tests we just wrote, they all share the same structure. In each test we set up an input value, call the function, and check the result. This three-part pattern has a name: Arrange — Act — Assert.

📋 The Structure of Every Test Case

In general, every test case in a suite follows the same three-part structure:

1  ARRANGE

Create any necessary preconditions for the test case, such as setting values of variables that affect the behavior of the SUT.

2  ACT

Exercise the SUT — call the method or trigger the behavior you are testing.

3  ASSERT

Verify that the result or behavior matches what was expected.

The assertion step makes each test Self-checking, eliminating the need for a human programmer to inspect test results. Now that we have built up all four tests one at a time, here they are together — each one annotated to show which phase is which:

📄 test_leap_year.py — complete test suite 
from calendar_utils import is_leap_year

# Category 1: not a multiple of 4
def test_not_multiple_of_4():
    # Arrange: year = 1973 (the input value is the only setup needed)
    # Act + Assert combined — trivial arrange means both fit on one line
    assert is_leap_year(1973) is False

# Category 2: multiple of 4, not of 100
def test_multiple_of_4_not_100():
    assert is_leap_year(2008) is True

# Category 3: multiple of 100, not of 400
def test_multiple_of_100_not_400():
    assert is_leap_year(1900) is False

# Category 4: multiple of 400
def test_multiple_of_400():
    assert is_leap_year(2000) is True

In this simple case the Arrange phase is trivial — there is no setup needed beyond the input value itself, so Arrange, Act, and Assert collapse onto a single line. In more complex tests (covered in Section 3) the three phases are clearly separated. Common assertion patterns in pytest:

On a value Example (pytest)
Value equality assert x == "Python"  /  assert x != "Python"
Boolean (truthy) assert x (non-False / non-None)
Regular expression assert re.match(r"YourRegexHere", s)
Object type assert isinstance(a, list)
On a block / call Example (pytest)
Exception raised with pytest.raises(ValueError): math.sqrt(-1)
Side effect before = len(reviews); delete_first_review(); assert len(reviews) == before - 1

One can invert the sense of any assertion (assert x != 50), but as Module 2 warned, negative assertions should be used with caution — there are many ways for a program not to satisfy a particular condition while still not behaving correctly.

3. Isolating Code: Doubles and Seams

We can distinguish three characteristics — which may occur individually or together — that complicate unit testing:

  1. The SUT relies on results from calling collaborator methods or invoking behaviors on collaborator objects
  2. The SUT has nondeterministic or time-dependent behavior
  3. The SUT has side effects
💡 In Plain English — Before We Dive In

All three reasons boil down to the same root problem: the SUT (System Under Test — the function or method you're testing) depends on something you don't fully control. A quick everyday translation of each:

  • Reason 1 — Collaborators: your function asks another function, object, class, or service to do part of the work for it — the same way you might ask a friend to look something up for you instead of knowing the answer yourself. If your friend's answer changes, so does yours.
  • Reason 2 — Nondeterminism / time: the same input doesn't always produce the same output — think of asking "what's the weather?" or checking "what time is it?" The answer today isn't the answer tomorrow.
  • Reason 3 — Side effects: the function reaches "outside itself" and changes the world — writing to a database, sending an email, printing to the screen — rather than simply returning a value.

The rest of Section 3 works through one running example that has reason 1 (a collaborator) and reason 2 (an unreliable network call baked into that collaborator), and shows exactly how we tame it for testing.

3.1 A Motivating Example: Looking Up a Movie

Suppose a controller action tries to search a remote database for one or more matches to a movie title:

  • The Movie model has a function find_in_tmdb that calls the API of an external service, The Movie Database (TMDb), and returns a list of Movie objects, which may be empty if there were no matches.
  • If there are no matches, the controller action should redirect the user back to the search page with an appropriate message.
  • If there is exactly one match, the controller should render a view that allows the user to enter a review for that movie.
  • If there is more than one match, the controller should render a different view that allows the user to specify which movie they want to review.
  • Because the model method relies on calling an external service, the call might fail. In that case, we assume find_in_tmdb will raise an exception ConnectionError.
🔄 Who's the SUT, and Who's the Collaborator?

Picture asking a friend for directions instead of knowing the route yourself:

You → ask your friend for directions → friend answers → you follow the answer

Here, you are the SUT (review_movie()), and your friend is the collaborator (Movie.find_in_tmdb(), which in turn talks to the real TMDb website over the Internet). The problem is exactly the same as with a real friend: you don't control what they say back. Today your friend might say "one match," tomorrow "no matches," and the day the Internet is down, they might not answer at all. That's precisely why review_movie() is hard to test as written — its behavior is only as predictable as find_in_tmdb()'s answer.

📄 views.py — review_movie 
def review_movie(request):
    search_string = request.GET.get("search")
    try:
        matches = Movie.find_in_tmdb(search_string)
        if not matches:  # nothing was found
            return redirect("review_movie", alert="No matches.")
        elif len(matches) == 1:
            movie = matches[0]
            return render(request, "review_movie.html", {"movie": movie})
        else:  # more than 1 match
            return render(request, "select_movie.html", {"movies": matches})
    except ConnectionError as err:
        return redirect("review_movie", alert=f"Error contacting TMDb: {err}")

Walking through it in plain terms: a user types a search string, review_movie() hands that string to its collaborator Movie.find_in_tmdb(), and depending on what comes back, the view takes one of four different paths:

1 Zero matches — find_in_tmdb() returns []

not matches is True (an empty list is "falsy"), so the view redirects back to the search page with the message "No matches."

2 Exactly one match — find_in_tmdb() returns a list of length 1

The elif len(matches) == 1 branch runs, movie = matches[0] grabs that single movie, and the view renders the review page so the user can write a review.

3 Several matches — find_in_tmdb() returns a list of length > 1

Neither of the above conditions is true, so the else branch runs and the view renders a selection page asking "Which movie did you mean?"

4 Network failure — find_in_tmdb() raises ConnectionError

Control jumps straight to the except block, and the view redirects back to the search page with an error message instead of a list of matches.

How would we unit-test this view? The Arrange step consists of preparing a request with some search string. The Act step consists of calling the view with that search string. But the Assert step depends on whether the call to find_in_tmdb returns an empty list, a list of exactly one match, a list containing more than one match, or raises an exception because of an error communicating with TMDb. There are really four test cases required here, and to test each of them, we essentially need to be able to control the behavior of the call to find_in_tmdb.

✅ The One Idea to Take Away From 3.1

We can't test all four branches reliably as long as find_in_tmdb() is free to talk to the real Internet. So the next question is: how do we take over — "force" — what find_in_tmdb() returns, on a test-by-test basis, without touching the source code of review_movie() itself? That's exactly what a seam lets us do, covered next.

3.2 Seams

🔑 Definition: Seam

Michael Feathers (2004) defines a seam as "a place where you can alter behavior in your program without editing in that place." In our case, we want to alter (control) the behavior of find_in_tmdb without changing the source code of the view function.

💡 In Plain English — A Seam is Like Swapping a Lightbulb

Imagine a light switch wired to a bulb:

Switch → Real Bulb   (normal operation)

Switch → Fake Bulb   (during testing)

The switch never changes — it flips the same way either time. Only the thing plugged in at the other end changes. That socket — the point where one thing can be swapped for another — is the seam. In our code, the line Movie.find_in_tmdb(...) is the socket: normally it's wired to the real TMDb-calling function, and during a test we temporarily wire it to a fake one instead. review_movie() keeps calling Movie.find_in_tmdb(...) exactly the same way either time — it never knows the difference.

The strategy is to temporarily replace find_in_tmdb so that instead of calling the real method, it calls a "fake" method whose behavior we control and can change for each test case. Such a construction is called a method stub. In pytest, this is most commonly done using the monkeypatch fixture or the unittest.mock library.

📄 test_views.py 
def test_redirects_to_search_if_no_match(client, monkeypatch):
    # Arrange: stub Movie.find_in_tmdb to return an empty list
    monkeypatch.setattr(Movie, "find_in_tmdb", lambda q: [])

    # Act
    response = client.get("/review_movie", {"search": "I Am Big Bird"})

    # Assert
    assert response.status_code == 302
    assert response.url == reverse("review_movie")
🔍 Breaking Down That monkeypatch.setattr(...) Line

It looks dense, but it's really just three pieces of information, in order:

Piece What it is Meaning here
Movie The class to patch "Replace something that lives inside the Movie class."
"find_in_tmdb" The attribute/method name "Specifically, replace the method called find_in_tmdb."
lambda q: [] The replacement function (the stub) "Whenever it's called with some argument q, just return [] — no matter what q is, and without ever touching the network."

lambda q: [] is nothing more than a short, inline way of writing def fake(q): return []. After this one line runs, every call to Movie.find_in_tmdb(...) anywhere inside review_movie() silently becomes a call to this fake function instead — no Internet connection is ever made.

The Arrange part of this test now includes setting up a stub for the method, specifying that when the stub is called, it should return an empty list — ensuring that not matches is true in our view, causing the redirect branch to execute next.

As is typical for a testing framework, monkeypatch automatically undoes the patch after each test function finishes, making the stub visible only to that test case and thereby keeping tests Independent. Note that it is not an error for find_in_tmdb to never be called — the stub setup only specifies what should happen if that method is called. If we wanted to express the test condition that the method must be called, we would use a Mock object and assert that it was called, as shown next.

3.3 Stubs vs. Mocks vs. Spies

📐 Test Doubles — A Vocabulary
  • Stub — a piece of code that replaces the real method's code with a controllable or fixed behavior for testing purposes. A stub only specifies what should happen if it's called.
  • Mock — a simplified "stunt double" of an object that can only mimic a few fixed behaviors, such as returning fixed values for specific attributes. Mocks are useful when a real object would be complex to instantiate because it has other dependencies, yet only a few specific properties of the object are necessary for the SUT to work properly. A mock additionally lets you assert that it was called.
  • Spy — similar to a stub, but lets a call to the real method proceed while "recording" the arguments and return values for later inspection.
  • Test double — the generic term covering all of the above and a few other types of seams.
💡 In Plain English — Same Idea, Everyday Words
  • Stub — a fake that just says "if you call me, here's what I'll say back." It doesn't care whether it actually gets called. Like leaving an answering machine message ready to play — it plays if someone calls, and nobody complains if nobody does.
  • Mock — a stub with a memory. It also says "and by the way, I'm keeping a receipt of every time I was called, and with what." You can later check that receipt: "Was I called? Exactly once? With 'Titanic'?"
  • Spy — lets the real method run as normal, but quietly stands next to it taking notes on the arguments and return value, so you can inspect them afterward without changing the real behavior at all.
  • Test double — the umbrella term for "any fake stand-in used during testing," the way "stunt double" covers any performer who stands in for the real actor.

A useful rule of thumb: use a stub when you only care about the return value the SUT receives; use a mock when it also matters whether/how the collaborator was actually called.

The term test double generically covers stubs, mocks, and spies. Figure 8.6 in the textbook summarizes typical strategies for using doubles in various unit-testing scenarios; the table below shows the same strategies translated to pytest.

System Under Test (SUT) Testing Strategy
Pure leaf function — no side effects, no collaborator methods or classes, same inputs yield same outputs Assert correct output results for critical values and for arbitrary values in noncritical regions
Relies on results from calling collaborator methods or invoking behaviors on collaborator objects In Arrange phase, create doubles that "force" the desired behavior by returning prearranged values, raising an exception, and so on
Nondeterministic or time-dependent behavior In code under test, isolate the nondeterminism in a method call that can be stubbed using a double in the Arrange phase
Has side effects In Arrange phase, observe the relevant state before executing test code; in Assert phase, observe it again and check for the side effect

Here are the four strategies above illustrated with concrete, everyday examples from a school grading app — a domain every student already understands. For each case the code under test is shown first so you can see exactly what is being tested, then the test itself:

📝 Case 1 — Pure leaf function: just call it and check

letter_grade(score) does pure arithmetic. It calls nothing else, so there is nothing to fake.

Code under test — grades.py
def letter_grade(score):
    if score >= 90: return "A"
    if score >= 80: return "B"
    if score >= 70: return "C"
    if score >= 60: return "D"
    return "F"
Test — test_grades.py
# Arrange: nothing needed — the input IS the setup
# Act + Assert in one line: call the function, check the result
def test_grade_for_90_is_A():
    assert letter_grade(90) == "A"

def test_grade_for_60_is_D():
    assert letter_grade(60) == "D"
☁️ Case 2 — Collaborator dependency: fake the thing you don't control

get_status_message() depends on weather.is_raining(), which calls a real weather API. We can't control whether it actually rains when the test runs, so we fake is_raining() to return exactly what we need.

Code under test — status.py
def get_status_message():
    if weather.is_raining():   # <-- calls external weather API
        return "Don't forget your umbrella!"
    return "Have a nice day!"
Test — test_status.py
def test_umbrella_reminder_when_raining(monkeypatch):
    # Arrange: replace weather.is_raining with a fake that always returns True
    # (we don't want the test to depend on actual weather!)
    monkeypatch.setattr(weather, "is_raining", lambda: True)

    # Act + Assert: call the function, check it picks the rainy message
    assert get_status_message() == "Don't forget your umbrella!"
📅 Case 3 — Implicit time dependency: freeze the clock

get_discount_percent() gives 50% off on Black Friday. If you run this test on any other day of the year, it would return 0 and the test would fail — even though the code is correct. So we freeze the date to Black Friday so the test always behaves the same way regardless of when it's run.

Code under test — shop.py
from datetime import date

def get_discount_percent():
    if date.today() == date(2024, 11, 29):  # Black Friday 2024
        return 50
    return 0
Test — test_shop.py
from datetime import date

def test_50_percent_discount_on_black_friday(monkeypatch):
    # Arrange: freeze date.today() to Black Friday — test always passes,
    # regardless of what day it actually is when you run the test
    black_friday = date(2024, 11, 29)
    monkeypatch.setattr(date, "today", classmethod(lambda cls: black_friday))

    # Act + Assert
    assert get_discount_percent() == 50
🏫 Case 4 — Side effects: check state before and after

enroll_student("Alice") doesn't return anything useful — its whole job is to change the classroom's state by adding a student to the roster. To test this, we record the class size before the call and verify it grew by exactly 1 afterward.

Code under test — classroom.py
roster = []

def student_count():
    return len(roster)

def enroll_student(name):
    roster.append(name)  # no return value — changes state as a side effect
Test — test_classroom.py
def test_enroll_increases_class_size_by_1():
    # Arrange: snapshot the count BEFORE the action
    before = student_count()

    enroll_student("Alice")  # Act: do the thing

    # Assert: the count AFTER must be exactly one more than before
    assert student_count() == before + 1
Case What makes it tricky Solution
1. letter_grade() — leaf function Nothing — it's entirely self-contained Just call it and check the return value

Asserting that a Mock Was Called

If we want to express the test condition that a method must be called — not just stub its return value — we use unittest.mock.Mock or MagicMock and assert on the call afterward:

from unittest.mock import patch

def test_find_in_tmdb_is_called_with_search_string():
    with patch.object(Movie, "find_in_tmdb", return_value=[]) as mock_find:
        client.get("/review_movie", {"search": "Milk"})

    # Assert: the stub MUST have been called with "Milk"
    mock_find.assert_called_once_with("Milk")
📝 Section Summary
  • When testing a method that has external dependencies — calling other methods or consuming other objects — we use test doubles to "stand in" for the real methods or objects and allow the test to tightly control the SUT's behavior.
  • Test doubles are set up in the Arrange phase of a test case. Stubs are set up to control return values from collaborator methods, while mocks are set up to mimic just those behaviors of the collaborator object used by the SUT.
  • Depending on what behavior is being tested, a test case can specify whether a particular stub must be called — that is, if the stub not being called signals a bug.
❓ Self-Check 8.3.1

Name two likely violations of FIRST that arise when unit tests actually call an external service as part of testing.

The test may no longer be Fast, since it takes much longer to call an external service than to compute locally. The test may no longer be Repeatable, since circumstances beyond our control could affect its outcome, such as the temporary unavailability of the external service.

4. Stubbing the Internet

When testing a method that makes a call to an external service via an API, there are many reasons we almost certainly don't want to make a real call to that API.

⚠️ A Real Cautionary Tale

Several years ago, the CS department head at a major US university received a complaint from a website that hosted academic papers, because a group of students had been working on a student project that repeatedly made "test" API calls against the real site. The site threatened to cut off the university's access if the students continued this behavior.

Another reason is that making real calls might prevent the test from being Repeatable (depending on how the remote service responds), and would almost certainly prevent the test from being Fast.

In fact, when testing our own app, all that we really care about is whether the API calls it would make are correctly formed — analogous to checking a call to a method stub to make sure the arguments are correct. So the more general question is: where should we stub external methods when testing an app that makes calls to an external service?

4.1 Stubbing "Close By" vs. "Far Away"

In Section 3 we chose to stub the Movie model itself and mimic the results of its calls to TMDb. But a more robust integration-testing approach would instead place the stub "closer" to the remote service. In particular, we could create fixtures — files containing the JSON content returned by actual calls to the service — and arrange to intercept calls to the remote service and return the contents of those fixture files instead.

🌐 Python Tooling: responses and VCR.py
  • The responses library does for Python what the Ruby gem Webmock does: it stubs out the entire web except for particular URIs that return a canned response when accessed by your code.
  • VCR.py (a port of Ruby's vcr gem) automates getting a response from the real service, saving the response data in a fixture file (called a cassette), and then "replaying" the cassette when your tests cause the remote service to be "called" — by intercepting low-level calls in Python's HTTP libraries.
📄 test_tmdb_integration.py — stubbing "far away" with responses 
import responses

@responses.activate
def test_find_in_tmdb_returns_one_match():
    # Arrange: intercept the TMDb URL and return canned JSON
    responses.add(
        responses.GET,
        "https://api.themoviedb.org/3/search/movie",
        json={"results": [{"title": "Milk", "id": 7508}]},
        status=200,
    )

    # Act
    matches = Movie.find_in_tmdb("Milk")

    # Assert
    assert len(matches) == 1
    assert matches[0].title == "Milk"

From an integration-testing standpoint, responses is the most realistic way to test interactions with a remote service, because the stubbed behavior is "farthest away" — we are stubbing as late as possible in the flow of the request. Therefore, when creating Behave/pytest-bdd scenarios to test external service integration, this approach is usually the right choice. From a pure unit-testing point of view (as in Section 3), it's less compelling, since we are concerned with the correct behavior of specific functions, and we don't mind stubbing "close by" in order to observe those behaviors in a controlled environment.

📝 Section Summary
  • To create Fast and Repeatable test cases for code that communicates with an external service, we use stubs to mimic the service's behavior. Test classes or fixtures can group tests that exercise different behaviors of the remote service.
  • The question of "where to stub" an external service depends on the purpose of the test. Stubbing "far away" using a library like responses is more realistic and appropriate for functional or integration tests; stubbing "close by" in the model or library that communicates with the remote service is often adequate for low-level unit tests.
❓ Self-Check 8.4.1

Is "stubbing the Internet" in conflict with the advice of Module 2 that one should avoid mocks or stubs in full-system Behave/pytest-bdd scenarios?

Full-system testing should avoid "faking" most parts of the application, as we have largely avoided doing using seams in this module. However, if the "full system" includes interacting with outside services we don't control, such as the interaction with TMDb in this example, we do need a way to "fake" their behavior for testing.

5. Fixtures and Factories

Doubles are appropriate when you need a stand-in with a small amount of functionality to isolate the code under test from its dependencies. But suppose you were testing a new method of class Movie called name_with_rating that returns a nicely formatted string showing a movie's title and rating.

Clearly, such a method would have to access the title and rating attributes of a Movie instance. You could create a double that knows all that information:

def test_name_with_rating_using_a_double():
    fake_movie = Mock(title="Casablanca", rating="PG")
    assert Movie.name_with_rating(fake_movie) == "Casablanca (PG)"

But since the method being tested is part of the Movie class itself, it makes sense to use a real object here, since this isn't a case of isolating the test code from collaborator classes.

5.1 Factories: Quickly Creating "Real" Objects

Where can we get a real Movie instance to use in such a test? Most testing frameworks for object-oriented languages support the use of factories — bits of code or declarative descriptions of objects designed to allow rapid creation of full-featured objects (rather than mocks) at testing time.

💡 The Goal of a Factory

The goal of a factory is to quickly create valid instances of a class using some default attributes that you can selectively override for testing. For example, if you were testing some code that allows a user to write a review for a movie, you might need a valid movie instance to pass to that code. You don't care what the movie's release date is, or who directed it — you just need a movie object that is valid and whose title and rating you do know.

Don't confuse this use of the term "factory" with the Abstract Factory design pattern discussed in Module 6 — they are unrelated concepts that happen to share a name.

You might think this seems like more work than just creating a movie instance directly by calling its constructor. In our simple example, that may be true. But there are two cases in which factories really shine:

  1. Objects with many required attributes. The object to be created has many attributes that must be initialized at creation time, even though any particular test case may only care about the specific values of a few of them. The app may have validations requiring a movie to have a valid release date or other fields meeting specific criteria, yet the test above doesn't care about the values of those other fields. In such cases, you can ask the factory to create an object in which certain attribute values are specified but others are filled in with valid defaults.
  2. Objects with relationships. Objects you need to create have has-many or belongs-to relationships with other objects, as Module 5 describes. For example, if a Review belongs to a Movie, and you are creating a set of tests to check various behaviors of a Review, you literally cannot create a valid Review instance without creating a Movie instance for it to belong to, even if the tests you are writing don't care about the movie itself. In this case, the Review factory can be configured so that creating a Review also creates a valid Movie to which it belongs.

5.2 Factory Boy: The Python Equivalent of FactoryBot

The Python library factory_boy lets you define a factory for any kind of model in your app and create just the objects you need quickly for each test, selectively overriding only certain attributes.

📄 tests/factories.py 
import factory
from datetime import timedelta
from django.utils import timezone
from .models import Movie


class MovieFactory(factory.django.DjangoModelFactory):
    class Meta:
        model = Movie

    title = "A Fake Title"          # default values
    rating = "PG"
    release_date = factory.LazyFunction(
        lambda: timezone.now() - timedelta(days=3650)  # 10 years ago
    )
📄 tests/test_movie.py 
from .factories import MovieFactory


def test_name_with_rating_includes_title_and_rating():
    # build() creates but doesn't save; create() also saves to the database
    movie = MovieFactory.build(title="Milk", rating="R")
    assert movie.name_with_rating() == "Milk (R)"

Using factories rather than fixtures preserves Independence among tests — each test builds exactly the object it needs, with no hidden dependency on shared state.

5.3 Fixtures: Pre-Loaded "Known Good" Objects

In database-backed apps, another source of "real" objects for use in tests is fixtures — a set of objects whose existence is guaranteed and fixed, and can be assumed by all test cases.

🏭 Where the Term Comes From

The term fixture comes from the manufacturing world: a test fixture is a device that holds or supports the item under test. Since all state in a database-backed app is kept in the database, a fixture file defines a set of objects that is automatically loaded into the test database before tests are run, so you can use those objects in your tests without first setting them up.

In pytest, the term "fixture" has a broader and more general meaning than the database-loading fixtures described above: a pytest fixture is any function decorated with @pytest.fixture that provides a piece of test setup (data, a connection, a temporary file, and so on) to the test functions that request it.

📄 conftest.py 
import pytest
from .factories import MovieFactory


@pytest.fixture
def hamilton(db):
    # Returns a known, valid Movie instance saved to the test database
    return MovieFactory.create(title="Hamilton", rating="PG-13")
📄 test_movie.py — using the fixture 
def test_hamilton_has_correct_rating(hamilton):
    assert hamilton.rating == "PG-13"

5.4 Use Fixtures Sparingly

⚠️ The Independence Tradeoff

Unless used carefully, fixtures can interfere with tests being Independent, as every test now depends implicitly on the fixture state — so changing the fixtures might change the behavior of tests. In addition, although any given test probably relies on only one or two fixtures, the union of fixtures required by all tests can become unwieldy. Therefore, fixtures should be used very sparingly if at all, and primarily for truly fixed data that, in production, would also be fixed and rarely change.

📝 Section Summary
  • When the SUT is itself an instance method (not a collaborator), use a real object rather than a double.
  • Factories (factory_boy in Python) let you create valid instances quickly, overriding only the attributes a particular test cares about, while filling in everything else with valid defaults. This is especially valuable for objects with many required attributes or with belongs-to/has-many relationships.
  • Database fixtures pre-load known-good data into the test database, but should be used sparingly because they create implicit dependencies across tests and can compromise Independence.
  • Factories generally preserve test independence better than fixtures, because each test builds exactly what it needs.

6. Coverage Concepts and Types of Tests

6.1 How Much Testing is Enough?

A poor but unfortunately widely-given answer is "As much as you can before the release deadline." A very coarse-grained alternative is the code-to-test ratio: the number of non-comment lines of code divided by the number of lines of tests of all types. In production systems, this ratio is usually less than 1 — that is, there are more lines of test than lines of app code.

Another widely-used metric, more conservative, is "when the rate of new bug reports falls below some threshold." This formulation acknowledges that while code can never be proven bug-free, bugs are getting harder to find. But a more precise way to approach the question is to combine such metrics with code coverage.

6.2 A Running Example

Consider this small Python class, which we'll use to illustrate coverage concepts:

📄 my_class.py 
class MyClass:
    def foo(self, x, y, z):
        if x:
            if y and z:
                self.bar(0)
        else:
            self.bar(1)

    def bar(self, x):
        self.w = x

6.3 The Coverage Hierarchy

Level Definition What it takes to satisfy it for MyClass
S0 — Method coverage Is every method executed at least once by the test suite? Call foo and bar at least once each
S1 — Call / Entry-Exit coverage Has each method been called from every place it could be called? Call bar from both inside foo's if branch and from its else branch
C0 — Statement coverage Is every statement of the source code executed at least once, counting both branches of a conditional as a single statement? In addition to calling bar, call foo at least once with x true (otherwise the statement that calls bar(0) is never executed), and at least once with y false
C1 — Branch coverage Has each branch been taken in each direction at least once? Call foo with both False and True values of x, and with values of y and z such that y and z evaluates once to True and once to False
C2 — Path coverage Has every possible route through the code been executed? In this example, with x, y, z treated as booleans, there are 8 possible paths
🔎 Decision Coverage — Stricter Than C1

A more stringent condition, decision coverage, requires that each subexpression that independently affects a conditional expression be evaluated to true and false. In this example, a test would additionally have to separately set y and z so that the condition y and z fails once for y being false and once for z being false.

🏆 Modified Condition/Decision Coverage (MC/DC)

MC/DC is a stricter standard used in aviation and safety-critical software (required by the FAA for flight software). In plain terms, it demands that for every boolean condition in a decision, you must have a test that shows that specific condition is the one making the difference — not some combination of conditions happening to produce the right result.

Example: for the condition if age >= 18 and has_id:, MC/DC requires at least one test where flipping only age >= 18 changes the outcome (to prove it matters independently), and another test where flipping only has_id changes the outcome. Formally it requires: every entry/exit point invoked at least once, every decision taking all possible outcomes at least once, and each condition independently shown to affect its decision's outcome.

6.4 How Much Coverage Is Achievable?

Achieving C0 coverage is relatively straightforward, and a goal of 100% C0 coverage is not unreasonable. Achieving C1 coverage is more difficult since test cases must be constructed more carefully to ensure each branch is taken at least once in each direction. C2 coverage is most difficult of all, and not all testing experts agree on the additional value of achieving 100% path coverage.

💡 The Real Value of Coverage Statistics

Code coverage statistics are most valuable to the extent that they highlight under-tested or untested parts of the code and show the overall comprehensiveness of your test suite — not as a "score" to chase to 100%.

6.5 Measuring Coverage in Python: coverage.py

In Ruby, the SimpleCov gem measures and displays C0 and C1 coverage. The Python equivalent is coverage.py, usually run via the pytest-cov plugin.

Install and run 
pip install pytest-cov

pytest --cov=myapp --cov-report=term-missing

This produces a report showing, file by file, what fraction of statements were executed (C0) and which specific line numbers were missed — letting you browse exactly which lines of your app were exercised by your test suite.

6.6 Unit, Functional, and System/Integration Tests

This module, and the above discussion of coverage, have focused on unit tests. Module 2 explained how user stories could become automated acceptance tests; we can think of a Behave/pytest-bdd scenario as both a system test (because it exercises code in many different parts of the application in the same way a user would) and an acceptance test (because a properly-written scenario reflects and verifies the behavior the user said they wanted).

🌐 Full-Stack Tests

In SaaS, such tests may also be called full-stack tests, since a typical scenario exercises every part of the app from the browser-based UI to the database. Unlike unit tests, system tests rarely rely on test doubles to isolate behavior; on the contrary, the goal is to simulate real users as closely as possible.

Any test that covers more than one method but is not a full-stack test is generically an integration test. For example, a pytest test of a Django view would probably stub out calls to the database and bypass the routing mechanism, neither of which is central to testing the view itself, but would probably include interactions with mechanisms such as parsing form input, which clearly are outside the view function itself.

System and integration tests are important, but insufficient. Their resolution is poor: if an integration test fails, it is harder to pinpoint the cause since the test touches many parts of the code. Especially for system tests, coverage also tends to be poor because even though a single scenario touches many classes, it executes only a few code paths in each class. For the same reason, system and integration tests also tend to take longer to run. On the other hand, while unit tests run quickly and can isolate the subject code with great precision — improving both coverage resolution and error localization — because they rely on fake objects to isolate the subject code, they may mask problems that would only arise in integration tests. In other words, high assurance requires both good coverage and a mix of all three kinds of tests.

Unit Functional System / Integration
What is tested One method / class Several methods / classes Large chunks of system
Running time Very fast Fairly fast Slow
Error localization Excellent Moderate Poor
Coverage Excellent Moderate Poor
Use of doubles Frequently Occasionally Rarely / never

6.7 Beyond Correctness: Other Flavors of Tests

We have focused on testing for correctness ("did you build the thing right"), but in practice, other flavors of tests are part of any comprehensive test suite:

  • Smoke test — a minimal attempt to operate the software, to see whether anything is obviously wrong before running the rest of the test suite. For example, if a low-level coding error prevents a SaaS app from displaying its home page or accepting logins, there is no point in running further tests.
  • Compatibility testing — less prominent in SaaS since the app developers control the server environment, but may still be important for testing the app's UI in different browsers. Services like SauceLabs support running SaaS integration tests on a variety of browsers and operating systems, and even capture a screencast of each run.
  • Regression testing — ensures that previously-fixed bugs do not reappear. (Covered further in Module 5.)
  • Performance, stress, and security testing — types of non-functional testing that ensure the software meets operational criteria, particularly important for SaaS. (Covered further in Module 7.)
  • Accessibility testing — ensures that the software is usable by persons with disabilities. In SaaS, accessibility testing focuses primarily on the client-side user experience.
📝 Section Summary
  • Static and dynamic measures of coverage — including code-to-test ratio, C0 or C1 coverage (reported by pytest-cov / coverage.py), and C2 coverage — measure the extent to which your test suite exercises different paths in your code.
  • Rather than setting "hard targets" for coverage levels, use coverage reports to identify under-tested parts of your app so you can enhance the test suite accordingly.
  • Unit, integration, and system/acceptance tests differ in terms of their running time, resolution (ability to localize errors), ability to exercise a variety of code paths, and ability to perform a "reasonableness check," or so-called smoke test, on the whole application. All three are vital to software assurance.
  • In addition to functional tests that check correctness, we also need non-functional tests for accessibility, compatibility, security, and performance.
❓ Self-Check 8.7.1

Why does high test coverage not necessarily imply a well-tested application?

Coverage says nothing about the quality of the tests. However, low coverage certainly implies a poorly-tested application.

❓ Self-Check 8.7.2

What is the difference between C0 code coverage and code-to-test ratio?

C0 coverage is a dynamic measurement of what fraction of all statements are executed by a test suite. Code-to-test ratio is a static measurement comparing the total number of lines of code to the total number of lines of tests.

7. Other Testing Approaches and Terminology

The field of software testing is as broad and long-lived as software engineering and has its own literature. Its range of techniques includes formalisms for proving things about coverage, empirical techniques for selecting which tests to create, and directed-random testing. Depending on an organization's "testing culture," you may hear different terminology than we've used in this module.

7.1 Covering a Graph of Possible Behaviors

One useful way to think about code is to divide it into basic blocks — each of which executes from beginning to end with no possibility of branching — and then join these basic blocks into a graph in which conditionals result in graph nodes with multiple out-edges. We can then think of testing as "covering the graph": each test case tracks which nodes in the graph it visits, and the fraction of all nodes visited at the end of the test suite is the test coverage.

One insight that emerges from this approach is that the levels of testing described in Section 6 refer to control flow coverage, since they are only concerned with whether specific parts of the code are executed or not.

7.2 Define–Use (DU) Coverage

🔢 Definition: DU-Coverage (Define-Use Coverage)

Every time a variable gets assigned a value, that is a define. Every time the variable's value is actually read and used, that is a use. DU-coverage asks: have your tests exercised every possible define-then-use pair?

Simple example:

discount = 0 # DEFINE (1st assignment)
if is_member:
    discount = 10 # DEFINE (2nd assignment)
apply_discount(discount) # USE

DU-coverage requires tests that exercise both possible define→use paths: one where is_member is False (discount stays 0 → used), and one where it is True (discount becomes 10 → used). A test suite that only tests is_member = True has incomplete DU-coverage — and might miss a bug in the default discount = 0 path.

7.3 Black-Box vs. White-Box Testing

Black-Box (Functional) Tests White-Box (Glass-Box) Tests
Design is based solely on the software's external specifications Design reflects knowledge about the software's implementation that is not implied by external specifications
Example: for a hash table, specify a random set of key/value pairs to test that storing then reading a key returns the stored value Example: exploit knowledge of the hash function to construct worst-case test data that results in many hash collisions; focus on boundary values

7.4 Mutation Testing

🧬 Definition: Mutation Testing

Invented by Ammann and Offutt, mutation testing is a technique where an automated tool makes small, syntactically legal changes to your source code — called mutations — and then runs your test suite against each mutated version. If a mutation causes no test to fail, your tests have a gap.

Example: Suppose your code is:

def is_adult(age):
    return age >= 18

A mutation tool might change >= to >, producing:

def is_adult(age):
    return age > 18 # mutated: >= changed to >

If none of your tests passes age = 18, this mutation survives — meaning your tests would not catch a real bug where someone aged exactly 18 is incorrectly denied adult access. Python's mutmut library automates this process.

7.5 Fuzz Testing

Fuzz testing consists of throwing random data at your application and seeing what breaks. In 2014, Google engineers reported that over a 2-year period, fuzz testing had helped find over 1,000 bugs in the open-source video-processing utility ffmpeg. Fuzz testing has been particularly useful for finding security vulnerabilities that are missed by both manual code inspection and formal analysis, including stack and buffer overflows and unchecked null pointers.

While such memory bugs do not arise in interpreted languages like Python, or in type-safe and memory-safe compiled languages such as Rust, fuzz testing can still find interesting bugs in SaaS apps:

  • Random / black-box fuzzing — either generates completely random data or randomly mutates valid input data, such as changing certain bytes of metadata in a JPEG image to test the robustness of the image decoder.
  • Smart fuzzing — incorporates knowledge about the app's structure and possibly a way to specify how to construct "realistic but fake" fuzz data. For example, smart-fuzzing SaaS might include randomizing the variables and values occurring in form postings or URIs, or attempting various cross-site scripting or SQL injection attacks (Module 7).
  • White-box fuzzing — uses symbolic execution, which simulates execution of a program, observing the conditions under which each branch is taken or not, then generates fuzzed inputs to exercise the branch paths not taken during the simulated execution. White-box fuzzing can theoretically provide C2 (all paths) coverage, but in practice the size of the search space is huge, and white-box fuzzing relies on a diverse set of "seed inputs" to be effective.

In Python, the hypothesis library brings property-based / fuzz-style testing into the pytest ecosystem, generating many randomized inputs to a function and checking that invariants hold across all of them.

📝 Section Summary

We can think of testing as "covering a graph" of possible software behaviors. The graph can represent control flow (basic block coverage), variable assignment and usage (DU-coverage), a space of random inputs (fuzz testing), or a space of possible tests with respect to specific errors in the code (mutation testing). The different approaches are complementary and tend to catch different types of bugs.

❓ Self-Check 8.8.1

The Microsoft Zune music player had an infamous bug that caused all Zunes to "lock up" on December 31, 2008. Later analysis showed that the bug would be triggered on the last day of any leap year. What kinds of tests — black-box, glass-box, mutation, or fuzz — would have been likely to catch this bug?

A glass-box test for the special code paths used for leap years would have been effective. Fuzz testing might have been effective: since the bug occurs roughly once in every 1460 days, a few thousand fuzz tests would likely have found it.

8. The Plan-and-Document Perspective on Testing

The project manager takes the Software Requirements Specification from the requirements planning phase and divides it into individual program units. Developers then write the code for each unit, and then perform unit tests to make sure they work. In many organizations, quality assurance staff perform the rest of the higher-level tests, such as module, integration, system, and acceptance tests.

8.1 Integration Strategies

There are three options for how to integrate the units and perform integration tests:

1 Top-down integration

Starts with the top of the tree structure showing the dependency among all the units. The advantage is that you quickly get some of the high-level functions working, such as the user interface, which allows stakeholders to offer feedback in time to make changes. The downside is that you have to create many stubs to get the app to limp along in this nascent form.

2 Bottom-up integration

Starts at the bottom of the dependency tree and works up. There is no need for stubs, as you can integrate all the pieces you need for a module. Alas, you don't get an idea how the app will look until you get all the code written and integrated.

3 Sandwich integration

Not surprisingly, tries to get the best of both worlds by integrating from both ends simultaneously. Thus, you try to reduce the number of stubs by selectively integrating some units bottom-up, and try to get the user interface operational sooner by selectively integrating some units top-down.

8.2 System Test and Validation

The next step for the QA testers after integration tests is the system test, as the full app should work. This is the last step before showing it to customers for them to try out. System tests cover both non-functional requirements, such as performance, and functional requirements of features found in the SRS.

In the plan-and-document process, the final test is for the customers to try the product in their environment to decide whether they will accept the product or not. That is, the aim is validation, not just verification. In Agile development, the customer is involved in trying prototypes of the app early in the process, so there is no separate system test before running the acceptance tests.

8.3 When Is Testing Complete?

One question for plan-and-document is how to decide when testing is complete. Typically, an organization will enforce a standard level of testing coverage before a product is ready for the customer. Examples might be statement coverage (all statements executed at least once), or all user input opportunities are tested with both good input and problematic input.

“

Program testing can be used to show the presence of bugs, but never to show their absence!

8.4 Formal Methods

Because testing alone can never prove the absence of bugs, there has been a great deal of research investigating approaches to verification beyond testing. Collectively, these techniques are known as formal methods. The general strategy is to start with a formal specification and prove that the behavior of the code follows the behavior of that spec. These are mathematical proofs, either done by a person or done by a computer:

  • Automatic theorem proving — uses mathematical logic rules to build a proof that the code is correct, the same way a mathematician proves a theorem step by step. Think of it as asking: "can I logically prove this code never crashes?" rather than just running tests hoping it doesn't.
  • Model checking — exhaustively checks every possible state the system can be in. For example, for a traffic-light controller with states (red, amber, green), it would automatically verify that the light can never show green in two directions at the same time — by trying every possible combination of states. Practical for small, well-defined systems like hardware controllers or network protocols; impractical for large apps because the number of states explodes.

📝 Section Summary
  • Testing and formal methods reduce the risks of errors in designs.
  • Unlike BDD/TDD, the plan-and-document process starts with writing code before you write the tests. Developers then perform unit tests.
  • Especially in large projects, different people perform the higher-level tests. Integration options for putting the units together are top-down, bottom-up, or sandwich.

  • Formal methods rely on formal specifications and automated proofs or exhaustive state search to verify more than what testing can do, but they are so expensive to perform that today they are only applicable to small, stable, critical portions of hardware or software.

9. Fallacies and Pitfalls

🚧 Fallacy: 100% Test Coverage with All Tests Passing Means No Bugs

There are many reasons this statement can be false. Complete test coverage says nothing about the quality of the individual tests. As well, some bugs may require passing a certain value as a method argument (for example, to trigger a divide-by-zero error), and control flow testing often cannot reveal such a bug. There may be bugs in the interaction between your app and an external service such as TMDb; stubbing out the service so you can perform local testing might mask such bugs.

⚠️ Pitfall: Dogmatically Insisting on 100% Test Coverage (All Green) Before You Ship

As we saw above, 100% test coverage is not only difficult to achieve at levels higher than C1, but gives no guarantees of bug-freedom even if you do achieve it. Test coverage is a useful tool for estimating the overall comprehensiveness of your test suite, but high confidence requires a variety of testing methods — integration as well as unit, fuzzing as well as hand-constructed test cases, define–use coverage as well as control-flow coverage, mutation testing to expose additional holes in the test strategy, and so on. Operational issues such as security and performance call for additional testing strategies beyond the correctness-oriented ones described in this module.

🚧 Fallacy: You Don't Need Much Test Code to Be Confident in the Application

While insisting on 100% coverage may be counterproductive, so is going to the other extreme. In production systems, the code-to-test ratio is usually less than 1 — that is, there are typically more lines of test code than lines of application code. A small test suite, even one with high statement coverage, is unlikely to exercise the variety of inputs, edge cases, and error conditions that a real application encounters in production.
